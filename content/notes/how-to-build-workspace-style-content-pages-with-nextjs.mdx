## 先说这个布局到底是什么

在很多博客系统里，内容页通常是：

- 上方标题
- 中间正文
- 侧边可选目录

而我在这个项目里采用的是另一种模式：**工作区式内容页**。

它的核心结构是：

1. 左侧目录
2. 中间正文
3. 右侧大纲

并且不是只在单一页面里这样做，而是把它变成了 `notes / projects / logs` 三个模块的统一模式。

这意味着页面不再只是“详情页”，而更像一个浏览工作区。

## 为什么要做成工作区，而不是普通列表页

一开始也可以走更常见的结构：

- 模块首页：列表
- 详情页：正文

但这种方案有一个明显问题：

用户进入模块之后，第一眼看到的是索引，而不是内容。

对于“个人站 + 内容系统”来说，这并不总是最优解。因为很多时候，我更希望用户进入模块后直接开始阅读，而不是先理解分类结构。

所以这里做了一个取舍：

- `/notes` 默认直接打开第一篇笔记
- `/projects` 默认直接打开第一个项目
- `/logs` 默认直接打开第一篇日志

这样，根路由本身就是工作区入口，而不是单纯的目录页。

## 这个布局的目标

工作区式布局主要想解决三件事：

### 1. 降低进入成本

用户点进模块后，不需要再多点一次才能开始读正文。

### 2. 保持结构感

左侧目录让整个模块始终有“内容全集”的感觉，而不是孤立地看一篇文章。

### 3. 提升长文可读性

右侧大纲让用户在长文中仍然知道自己读到哪里，以及接下来还有什么内容。

## 路由层怎么设计

这个布局的第一步并不是 CSS，而是路由设计。

### 根路由默认打开第一篇

以笔记模块为例，根页会直接选择 `notes[0]` 作为当前内容。

这样 `/notes` 本身就能渲染完整的三栏工作区，而不是先显示一个空列表页。

### 详情页复用同一个工作区组件

而 `/notes/[slug]` 也不会切换成另一套详情页布局，而是继续复用同一个工作区组件，只是把选中的内容换掉。

项目和日志模块也是同一个思路。

这套做法的好处很明显：

- 根页和详情页体验一致
- 组件复用更自然
- 内容浏览路径更顺

## 工作区组件的最小结构

无论是笔记、项目还是日志，工作区组件的结构都大致相同：

```tsx
<div className="page-shell">
  <section className="workspace-grid">
    <aside className="left-nav">目录</aside>
    <div className="main-content">正文</div>
    <aside className="right-toc">大纲</aside>
  </section>
</div>
```

真正的关键，不是结构写出来，而是这三栏如何在桌面端和移动端表现得自然。

## 三栏布局的核心逻辑

这个项目里三栏布局采用的是桌面端网格：

- 左栏固定宽度
- 中间正文自适应
- 右栏固定宽度

例如桌面端会用类似下面的网格思路：

```tsx
xl:grid-cols-[12rem_minmax(0,1fr)_13rem]
```

这里中间使用 `minmax(0,1fr)` 非常关键，因为它能避免正文区域在长内容或宽内容时把整体网格撑坏。

## 为什么左右两栏要用 sticky

如果只是普通三栏流式布局，当页面滚动时，左右栏会和正文一起移动，很快就失去“辅助导航”的意义。

所以桌面端要做两件事：

### 1. 左侧目录 sticky

用户滚动正文时，始终能切换到同模块的其他内容。

### 2. 右侧大纲 sticky

用户阅读长文时，始终能看到当前文章的章节结构。

这就是为什么左右栏不是普通容器，而是：

```tsx
xl:sticky xl:top-24 xl:self-start
```

`top-24` 的本质，是为顶部 sticky header 留出安全距离。

## 中间正文为什么不做“内部独立滚动”

这是这类布局里最容易走歪的地方。

很多人看到“左右固定、中间阅读”时，第一反应会把正文做成一个固定高度、内部滚动的容器。

这样表面上看好像很像应用工作区，但问题很多：

- 鼠标滚轮焦点不稳定
- 页面整体滚动行为变差
- 浏览器回退与滚动记忆体验变差
- 可访问性也会变差

因此这个项目后来采用的是更稳的方案：

- 页面保持正常文档流滚动
- 左右栏使用 sticky 固定
- 只有当侧栏内容过高时，才让侧栏内部滚动

这会让最终体验更接近“正常网页 + 固定辅助导航”，而不是“页面里再嵌一个应用窗口”。

## 左侧目录怎么做

左侧目录承担的是“模块内部导航”，而不是文章内部导航。

比如：

- 笔记目录显示所有笔记
- 项目目录显示所有项目
- 日志目录显示所有日志

当前选中项会高亮，并通过小圆点强化状态感。

这种目录的价值在于，用户始终知道：

- 当前模块里还有哪些内容
- 自己当前看的是哪一篇

这比单纯的“上一篇 / 下一篇”更有知识库感。

## 右侧大纲怎么做

右侧大纲承担的是“文章内部导航”。

大纲来源不是手写，而是内容编译阶段从标题中自动提取出来的结构化数据。

所以页面层只需要消费 `headings` 数组，然后渲染成锚点链接即可。

这带来两个好处：

### 1. 正文与大纲天然同步

不用额外手工维护目录。

### 2. 大纲可以跨模块复用

只要正文有标题结构，项目、笔记、日志都能复用同一套大纲逻辑。

## 移动端怎么处理

桌面端三栏很自然，但移动端不可能照搬。

在小屏里，最合理的做法通常是：

- 隐藏右侧固定大纲
- 把大纲折叠成 `details`
- 左侧目录回到正文下方或正文前方的普通块级区域

这个项目里用的是比较轻的方式：

- 桌面端显示右侧大纲
- 移动端用 `details` 展开大纲

这样既保留了信息结构，也不会让移动端布局过于拥挤。

## 项目、笔记、日志为什么分成三个 workspace 组件

从抽象角度看，这三个组件其实已经非常相似了。

那为什么没有一开始就抽成一个通用 `ContentWorkspace`？

原因主要有两个：

### 1. 第一版优先跑通结构

项目、笔记、日志虽然布局相似，但头部信息并不相同：

- 项目有 role / status / stack / repo / demo
- 笔记有 category / tags
- 日志有 period / tags

过早抽象，反而会让组件接口变复杂。

### 2. 保留局部演化空间

第一版阶段，更重要的是让每个模块先形成稳定模式，而不是过度追求代码收敛。

这是一种典型的工程取舍：

- 短期接受重复
- 换取实现速度和可读性
- 后期再统一抽象

## 页面级背景为什么也是布局的一部分

工作区布局不只是结构问题，也涉及页面氛围。

这个项目里：

- 首页是浅蓝 / 深蓝的品牌感背景
- 项目、笔记、日志是更偏暖的内容工作区背景
- 关于页则介于品牌页和资料页之间

这意味着“布局”和“页面背景语言”是耦合的。

工作区并不是简单三栏，而是一整套页面层材质设计：

- 左栏是轻材质导航卡
- 中间是重点正文卡
- 右栏是极简大纲栏
- 背景负责营造整个模块的气质

## 这套布局适合什么内容站

我认为它特别适合：

- 技术博客
- 项目案例站
- 个人知识库
- 课程型文档站

尤其适合“内容之间关系很强”的场景，因为左侧目录会不断提醒用户：当前阅读只是整个模块中的一个节点。

## 这套布局不适合什么场景

也不是所有站点都适合工作区式布局。

比如这些场景就未必合适：

### 1. 单篇文章为核心的资讯站

如果用户主要从搜索引擎进入单篇内容，目录式工作区未必比普通详情页更有价值。

### 2. 内容量特别大但筛选复杂

如果条目非常多，左侧目录会迅速失去效率，这时更需要搜索、筛选和分组系统。

### 3. 以卡片瀑布流为主的视觉站

这类站点更强调总览，不一定需要固定三栏阅读结构。

## 这套实现里最值得继续优化的点

当前这套方案已经稳定，但还有明显的优化空间。

### 1. 抽象通用 Workspace 组件

现在三个模块结构高度相似，后续可以抽一层统一组件。

### 2. 当前项联动高亮

右侧大纲目前主要是静态锚点导航，后续可以增加基于滚动位置的激活状态同步。

### 3. 更强的目录分组能力

左侧目录未来可以支持按分类、年份或标签分组。

### 4. 搜索与筛选

当内容量上来后，仅靠左侧目录会不够，需要进一步引入搜索。

## 最后总结

用 Next.js 做工作区式内容页，关键不在于把页面拆成三栏，而在于想清楚：

- 根路由是不是直接进入内容
- 左侧目录到底服务什么导航
- 右侧大纲到底服务什么阅读行为
- 页面应该保持正常滚动，还是内部滚动
- 桌面端和移动端如何分层处理

这个项目给出的答案是：

- 根页直接打开第一篇
- 左侧做模块目录
- 右侧做文章大纲
- 页面保持自然滚动
- 桌面端三栏固定，移动端折叠大纲

这让内容页不再只是“详情页”，而是一个真正可浏览、可切换、可定位的工作区。
