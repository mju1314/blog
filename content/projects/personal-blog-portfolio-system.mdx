## 项目背景

这个项目最开始并不是为了“做一个博客”，而是为了回答一个更实际的问题：

当一个开发者同时想做技术沉淀、项目展示、阶段复盘和求职表达时，应该用什么样的信息结构把这些内容统一起来，而不是把它们拆成彼此割裂的页面。

传统个人站通常会落入两种模式之一：

- 只做博客，文章很多，但项目表达弱，难以快速体现工程能力
- 只做作品集，项目截图很多，但缺少过程、方法和持续输出

这个项目的目标，是把这两种方向合并为一套内容驱动的个人站系统：

- 首页负责建立第一印象与分发入口
- 笔记模块负责沉淀技术文章与学习材料
- 项目模块负责展示案例、取舍与实现思路
- 日志模块负责记录阶段推进、周记与复盘
- 关于页负责承担简历式概览
- 留言页负责提供最低成本的互动入口

从产品定义上看，它更接近“个人品牌站”，而不是单一博客。

## 项目目标

第一版的目标并不复杂，但要求边界清晰：

1. 内容必须可持续维护，而不是依赖后台系统才能更新
2. 项目、笔记、日志三类内容要能被统一组织，但又保留各自语义
3. 页面结构不仅要能展示内容，还要能够直接承担求职和对外展示功能
4. 视觉上要避免模板感，形成相对完整的个人站风格
5. 站点必须具备基础发布能力，包括深色模式、SEO、sitemap、robots 和留言能力

因此，这个项目的核心不是“功能堆叠”，而是通过一组相互配合的结构，形成稳定的内容生产与展示系统。

## 整体方案

整个系统可以拆成下面几层：

1. 内容建模层
2. 内容编译层
3. 数据聚合层
4. 路由与页面层
5. 工作区布局层
6. 视觉主题层
7. 发布与扩展层

这样拆开的目的，是让项目既能服务当前页面展示，也能在后续继续扩展。

## 为什么选择 MDX 而不是 CMS

这个项目没有选择 headless CMS，也没有引入数据库，而是采用了 `MDX + content-collections` 的内容驱动方案。

这样做的原因主要有三个：

### 1. 当前阶段的内容量可控

第一版内容并不多，完全没必要为了少量内容引入一整套后台、数据库和同步逻辑。

### 2. 内容和代码天然耦合

项目页、技术笔记和日志页本身就需要：

- 代码块
- 标题结构
- 标签和 frontmatter
- Git 版本管理

这些能力本来就和 Markdown / MDX 非常匹配。

### 3. 降低系统复杂度

用文件系统做内容源，意味着第一版不需要处理：

- 后台权限
- 数据迁移
- 接口同步
- 管理端维护

对于一个个人站来说，这是更符合成本收益比的方案。

## 内容系统设计

这个项目的内容系统基于 `content-collections.ts` 统一定义。

它不是简单地“读取 Markdown 文件”，而是在构建期完成了几件关键工作：

- frontmatter schema 校验
- MDX 编译
- 标题提取
- 阅读时长计算
- 统一生成可消费的数据结构

### 基础字段抽象

三类内容共享一组基础字段：

- `title`
- `summary`
- `date`
- `updated`
- `slug`
- `draft`
- `featured`
- `tags`
- `content`

这保证了内容系统在扩展时仍然有统一的数据骨架。

### 差异化字段

在共享字段之外，每类内容有自己的业务字段：

#### 笔记

- `category`
- `toc`

#### 项目

- `role`
- `status`
- `stack`
- `repo`
- `demo`

#### 日志

- `period`

这种设计避免了“所有内容都强行当文章处理”的问题，也让每个模块更符合自身语义。

## 编译链路

内容编译的核心目标，是把“原始 MDX 文件”转换成“页面层可以直接消费的数据”。

这里主要做了四件事：

### 1. MDX 编译

通过 `compileMDX` 把正文编译成可渲染的内容结构。

### 2. 标题提取

系统会扫描二级和三级标题，生成目录大纲，用于页面右侧导航和移动端折叠目录。

### 3. 阅读时长计算

通过对文本进行简单分词估算阅读时间，并排除代码块内容，避免阅读时长偏差过大。

### 4. 发布时间标准化

把日期转换成 `publishedAt`，用于后续统一排序。

这一步很重要，因为它把内容层从“纯文本文件”提升成了“具备结构化元信息的页面数据”。

## 数据聚合层

内容编译完成后，并不会在页面里直接使用原始集合，而是进入一层统一的聚合逻辑。

这里主要做了两件事：

### 1. 草稿过滤

在开发环境可以看到草稿，在生产环境会自动过滤 `draft: true` 的内容。

### 2. 时间排序

三类内容都按发布时间倒序排列，保证列表与默认展示逻辑一致。

此外，笔记还额外提供了：

- 分类聚合
- 标签聚合
- 按分类筛选
- 按标签筛选

这意味着页面层拿到的已经不是底层内容集合，而是接近“查询接口”的结果。

## 页面与路由设计

这个项目的一个关键设计是：`notes / projects / logs` 都不是传统意义上的列表页，而是工作区入口页。

### 默认打开第一篇

三个模块的根路由都采用同一种策略：

- `/notes` 默认打开第一篇笔记
- `/projects` 默认打开第一个项目
- `/logs` 默认打开第一篇日志

这样做的好处是，用户进入模块后不会停留在“空入口”或“纯索引页”，而是直接进入可阅读内容。

### 详情页复用同一套工作区

`/notes/[slug]`、`/projects/[slug]`、`/logs/[slug]` 并不是另一套独立详情页，而是继续复用工作区组件，只是切换选中的内容。

这个设计让根页面和详情页的体验保持一致，也降低了维护成本。

## 工作区布局设计

这是这个项目最值得单独展开的一部分。

三个内容模块都采用了统一的三栏工作区结构：

1. 左侧目录
2. 中间正文
3. 右侧大纲

### 左侧目录

左侧负责展示同一模块下的全部内容入口，例如：

- 所有笔记
- 所有项目
- 所有日志

桌面端采用 `sticky` 定位，保持目录稳定；当目录内容超出视口高度时，再在侧栏内部滚动。

### 中间正文

中间正文是主阅读区域，承载：

- 标题
- 摘要
- 元信息
- 标签
- 正文 MDX

它在视觉上被处理为大卡片，以便与左右导航形成层次分离。

### 右侧大纲

右侧通过标题提取结果生成大纲，桌面端固定显示，移动端折叠成 `details` 面板。

这种设计可以兼顾长文阅读和小屏适配。

### 为什么不用传统“列表页 + 详情页”

因为当前项目更强调“浏览过程中的连续性”，而不是单篇文章跳转体验。

工作区模式的优势在于：

- 更适合知识库式浏览
- 目录和正文关系更紧密
- 进入成本更低
- 模块感更强

但代价是：首页之外的内容入口会更像“应用工作区”，不再是传统博客的索引结构。

## 首页设计思路

首页承担的不是简单导航，而是整站的价值表达。

首页内容被拆成几组：

### 1. Hero 区

用于建立第一印象，表达站点定位和个人技术方向。

### 2. 能力卡片区

把站点内容方向拆成几个专题模块，例如：

- 技术文章
- 项目专栏
- Agent 实验
- 中间件 / 工程化
- 前端与交互
- 日志与复盘

它们本质上既是内容入口，也是“我擅长什么”的视觉表达。

### 3. 精选项目区

展示最有代表性的项目，而不是平铺所有项目。

### 4. 最新笔记与近期日志

用来体现内容更新节奏和长期输出能力。

因此，首页不是简单的门户，而是一个兼顾品牌表达和内容分发的复合页面。

## 视觉系统设计

这个项目在视觉上做了两层设计：

### 1. 全局主题变量

在全局样式中定义语义色变量，例如：

- `background`
- `foreground`
- `surface`
- `border`
- `accent`

这样做的好处是：

- 深色模式切换更简单
- 页面之间风格可统一
- 后续改版不会大面积改组件代码

### 2. 页面级背景语言

不同模块使用不同页面根类来切换背景与氛围，例如：

- `home-page`
- `about-page`
- `project-page`
- `note-page`
- `log-page`

这让站点在统一主题下仍能形成模块差异，而不是所有页面完全一个底色。

## 深色模式设计

深色模式不是简单反色，而是重新定义一套适合阅读和展示的深色材质：

- 降低高亮白色区域
- 保持足够的文字对比度
- 调整卡片边框、阴影和高光层
- 保证首页与 about 页在深色下的氛围一致

近期对 about 页的调整，本质上也是在解决“深色模式下白色高光过强”的问题，使其更接近首页的深色视觉语言。

## 关于页的角色

这个项目里的 about 页不是传统“自我介绍页”，而是轻量版简历。

它承担的职能包括：

- 说明当前职业方向
- 展示技能栈分组
- 给出后续规划与定位

这让整个站点从“有内容”进一步变成“有职业表达能力”的系统。

## 留言页设计

这个项目没有在每篇文章中都集成评论，而是只保留了一个统一留言页。

这样做的原因是：

- 第一版交互成本更低
- 配置更简单
- 审核和维护更容易
- 依然保留了公开互动入口

Giscus 的接入方式也保持了最小实现：

- 通过环境变量读取配置
- 未配置时显示缺失提示
- 主题跟随站点深色模式切换

这是一个很典型的“用最少成本保留产品闭环”的实现。

## SEO 与发布能力

虽然它是个人项目，但在发布能力上已经具备基础完整性：

- 根布局中配置 metadata
- 支持 Open Graph / Twitter metadata
- 自动生成 sitemap
- 自动生成 robots
- 所有内容页使用静态生成参数

这些能力保证它不是一个只能本地展示的 demo，而是可以直接部署上线的站点。

## 当前做过的重要取舍

这个项目里有几个取舍是有意识做出的。

### 1. 不引入 CMS

为了降低系统复杂度，让第一版更快落地。

### 2. 不引入数据库

因为当前内容模型天然更适合文件系统管理。

### 3. 不做文章级评论

统一留言页的维护成本更低，更适合第一版。

### 4. 工作区优先于索引页

更强调内容阅读和结构化浏览，而不是传统博客式列表浏览。

### 5. 多组件重复优先于过早抽象

目前 `ProjectWorkspace`、`NoteWorkspace`、`LogWorkspace` 结构高度相似，但没有在第一时间做成一个极度抽象的通用组件。

这样做的优点是：

- 开发阶段更快
- 结构更直观
- 更容易按模块单独演化

代价则是：

- 代码重复
- 后续重构空间较大

## 当前不足

这个项目已经形成了完整骨架，但仍有一些明显的后续空间：

### 1. 三个工作区组件可继续抽象

它们已经形成清晰模式，后续可以收敛成更通用的 `ContentWorkspace`。

### 2. 标题提取逻辑较轻量

目前主要基于 Markdown 行解析，后续可以升级为更稳健的 AST 级处理。

### 3. 内容发现能力仍可增强

当前工作区模式适合阅读，但分类导航、搜索、筛选还可以继续补强。

### 4. 真实内容仍需持续替换

目前骨架已经完整，但长期价值最终还是取决于持续更新的真实内容。

## 最终结果

这个项目最终交付的不是一个“页面拼完了的前端作品”，而是一套已经能够实际承载个人内容、项目表达和职业展示的内容系统。

它的价值主要体现在三点：

1. 能长期沉淀内容
2. 能结构化展示工程能力
3. 能作为个人品牌与求职入口持续演进

如果把它当成第一个正式项目来看，它最有意义的地方并不是“做了多少炫技功能”，而是把信息架构、内容模型、页面结构、视觉系统和发布能力连成了一套完整方案。

这也意味着它可以继续向两个方向发展：

- 作为个人长期维护的内容站
- 作为后续技术文档和项目复盘的母体项目
